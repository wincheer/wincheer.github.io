Git模块二：本地仓库核心操作

欢迎回来！在模块一，我们成功安装并配置了 Git，相当于为我们的“作家”生涯办好了“作者证”。现在，我们要正式开始创作了。

在这个模块，我们将学习如何开启一本新书（一个新项目），如何将我们的草稿（代码文件）一章一章地整理、存档，并回顾我们的创作历史。我们会接触到 Git 中最高频、最核心的一组命令。

准备好了吗？让我们把一个普通的文件夹，变成一个拥有超能力的“智能项目”吧！

【场景】

想象一下，你接到了两个不同类型的任务：

  1. 开启一个全新的项目：领导让你负责一个全新的内部工具，代号“Phoenix”。目前它还只存在于你的想法里，你需要从零开始创建项目文件夹，写下第一行代码。

  2. 加入一个已有的项目：你被分配到了公司的主力产品“Ocean”项目组。这个项目已经有成千上万行代码，有几十位同事在共同开发。你需要把整个项目完整地拷贝到你的电脑上，以便开始你的开发工作。

这两个场景代表了我们日常工作的两种起点。我们怎么用 Git 来优雅地处理这两种情况呢？我们如何开始对一个项目进行版本“存档”？我们又如何知道哪些文件已经被“存档”，哪些还是“草稿”？

这些问题，正是本模块要解决的核心痛点。

【理论】

1. 什么是 Git 仓库？—— 项目的“记忆核心”

在我们运行任何命令之前，我们必须先回答这个“第一性问题”：我们说的“仓库”或“版本库”（Repository）到底是什么？

当你对一个文件夹执行了“初始化”操作后，Git 会在这个文件夹里创建一个隐藏的子文件夹，名字叫 .git。

比喻：这个 .git 文件夹，就是你这个项目（游戏存档）的“记忆核心”或“大脑”。

它里面包含了关于你项目版本的所有信息：

  • 所有的“存档点”（Commits）

  • 完整的历史记录（Logs）

  • 分支信息（Branches）

  • 还有其他所有让 Git 正常运转的元数据。

把它想象成飞机的黑匣子。你绝对、绝对不要手动去修改 .git 文件夹里的任何东西！ 除非你真的知道自己在做什么，否则任何手动更改都可能导致你的版本历史（也就是全部心血）彻底损坏。我们只通过 git 命令与它安全地交互。我们只需要知道，它的存在，就意味着这个文件夹已经被 Git 所管理。我们后续的所有 git 命令，本质上都是在和这个 .git 文件夹沟通。删掉它，版本历史就灰飞烟灭了。

2. 文件的生命周期 —— 从“新想法”到“永久存档”

这是 Git 本地操作中最核心的理论。在模块一我们了解了四个区域，现在我们聚焦于前三个区域，看看一个文件是如何在其中流转的。

一个文件有四种主要状态，理解了这个流程，git status 命令的输出对你来说就会像读故事一样简单。

  1. Untracked (未追踪)

    • 含义: 这是一个新文件，Git 已经注意到了它的存在，但还没有将它纳入版本控制。

    • 比喻: 你在书桌上刚写下的一张新的便签纸。它就在那里，但你还没决定要不要把它收录进你的书稿里。

  2. Tracked - Unmodified (已追踪 - 未修改)

    • 含义: 文件已经被 Git “存档”，并且自从上次存档后，你没有对它做任何改动。

    • 比喻: 已经正式出版并存入你个人书库的某一章节。它是干净的、已归档的状态。

  3. Tracked - Modified (已追踪 - 已修改)

    • 含义: 文件已经被 Git “存档”过，但是你现在又对它进行了修改。

    • 比喻: 你从书库里拿出了已出版的某一章，用红笔在上面做了些修改和批注。它和书库里的版本已经不一样了。

  4. Staged (已暂存)

    • 含义: 你已经明确告诉 Git，你希望将当前这个版本的修改包含在下一个“存档点”中。

    • 比喻: 你把你修改好的那一章，郑重地放进了桌上的 “待出版”文件夹，等待和其他章节一起付印。

3. .gitignore 文件 —— 项目的“忽略清单”

第一性问题：是不是项目里的所有文件我们都想进行版本控制？

当然不是。设想一下，你的项目可能会自动生成一些日志文件 (.log)，或者在编译过程中产生大量的临时文件 (比如 node_modules 文件夹)，或者包含一些你不希望上传的本地配置文件（比如包含密码的 config.local.js）。

把这些文件都存进 Git 仓库会是一场灾难：

  • 仓库会变得异常臃肿。

  • 会暴露敏感信息。

  • 会记录下大量无意义的变更。

.gitignore 文件就是用来解决这个问题的。它是一个纯文本文件，你可以在里面列出不希望 Git 去追踪的文件或文件夹的匹配模式。

比喻：.gitignore 就是你给你的图书管理员（Git）的一张“忽略清单”，告诉他：“这些类型的草稿、笔记和废纸，你永远都不用管，也不用给我存档。”

【核心】

好了，理论学习完毕，让我们进入实战演练。我们将完整地走一遍从创建项目到提交第一个存档点的全过程。

1. 创建一个新仓库 (git init)

这对应我们前面提到的“Phoenix”新项目场景。

  • 逐行精讲:

    • mkdir phoenix-project: 创建一个名为 phoenix-project 的文件夹。(mkdir 是 make directory 的缩写)

    • cd phoenix-project: 进入这个文件夹。(cd 是 change directory 的缩写)

    • git init: 这是我们今天学的第一个核心命令。它告诉 Git：“请把当前这个文件夹变成一个由你管理的仓库吧！”。执行后，你会看到类似 Initialized empty Git repository in .../.git/ 的提示，并且一个隐藏的 .git 文件夹会被创建。

2. 查看文件状态 (git status)

git status 是你在 Git 中最重要的朋友。当你感到迷茫，不知道当前项目处于什么状态时，运行 git status 就对了！

此时，输出会告诉你 On branch master, No commits yet, nothing to commit。这很正常，因为我们的仓库是空的。

3. 创建、添加与提交文件

现在，我们来模拟一次完整的工作流程：写代码 -> 暂存 -> 提交。

  • 逐行精讲:

    • echo "# Phoenix Project" > README.md: 创建一个名为 README.md 的文件，并把 # Phoenix Project 这行文字写进去。

    • git status: 这次运行，你会看到一个非常重要的变化！Git 会提示 Untracked files: 并列出 README.md。这完美印证了我们理论部分的“文件生命周期”，这个新文件正处于 Untracked (未追踪) 状态。

  • 逐行精讲:

    • git add README.md: 告诉 Git：“我希望把 README.md 这个文件的当前内容，放进‘待出版’文件夹（暂存区），准备进行下一次存档。”

      友情提示： 除了指定单个文件名，你还可以使用 git add . (注意有个点)，这个命令会把当前目录下所有“未被忽略的”新文件和被修改过的文件一次性全部添加到暂存区。这在你修改了多个文件时非常方便。但请确保你清楚地知道自己添加了哪些文件，git status 会是你的好帮手！

    • git status: 再次运行，你会看到提示变成了 Changes to be committed:，下面列着 new file: README.md。文件状态已经从 Untracked 变成了 Staged (已暂存)。

  • 逐行精讲:

    • git commit: 告诉 Git：“请把暂存区里所有的东西打包，创建一个永久的存档点吧！”

    • -m: 是 -message 的缩写，表示我们要直接在命令行里提供本次存档的说明。

    • "feat: Initial commit with README": 这就是存档说明。我们遵循公司的 Conventional Commits 规范。feat: 表示这是一个新功能（feature）类型的提交。

    • git status: 运行后，你会看到 nothing to commit, working tree clean。这表示，你的书桌（工作区）上的内容，和你个人书库（本地仓库）里的最新版本，是完全一致的。文件状态现在是 Tracked - Unmodified (已追踪 - 未修改)。

4. 查看提交历史 (git log)

我们已经创建了一个存档点，怎么回头看呢？

  • 输出解读:

    • commit <一长串字符>: 这是这次提交的唯一 ID，就像是它的身份证号。

    • Author: 本次提交的作者（就是你在模块一配置的用户名和邮箱）。

    • Date: 提交的时间。

    • feat: Initial commit with README: 你写的提交说明。

5. 克隆一个现有仓库 (git clone)

这对应我们前面提到的“Ocean”老项目场景。这个操作要简单得多。

  • 逐行精讲:

    • git clone: 告诉 Git：“请帮我把远程中央图书馆的某本书，连同它的全部历史，完整地复制一份到我的电脑上。”

    • https://github.com/company/ocean-project.git: 这就是远程仓库的地址。这个 URL 通常可以在项目 GitHub 页面的“<> Code”按钮那里找到。

    • 这个命令会自动创建一个 ocean-project 文件夹，并在里面包含 .git 记忆核心和项目的所有文件。一步到位，非常方便。

【实践】

现在，轮到你来当一次“作家”，为自己的小项目创建几个“存档点”了。

任务目标: 练习完整的本地 add -> commit 流程，并学会使用 .gitignore。

操作步骤:

  1. 创建项目:

    • 在你的电脑上创建一个新的文件夹，叫做 my-first-repo。

    • 进入这个文件夹。

    • 使用 git init 命令将它初始化。

  2. 第一次提交:

    • 创建一个 index.html 文件，可以随便写点内容，比如 <h1>My Website</h1>。

    • 运行 git status 查看文件状态（它应该是 Untracked）。

    • 使用 git add index.html 将它暂存。

    • 再次运行 git status 查看状态（它应该变成了 Staged）。

    • 使用 git commit -m "feat: Create initial index.html" 提交你的第一个存档点。

  3. 第二次提交与忽略文件:

    • 创建一个 styles/ 文件夹，并在其中创建一个 main.css 文件。

    • 创建一个名为 temp.log 的临时文件。

    • 关键一步: 创建一个 .gitignore 文件。

    • 在 .gitignore 文件里，新起一行，写入 *.log (这表示忽略所有以 .log 结尾的文件)。

    • 现在运行 git status。你会发现 Git 只提示你 styles/ 和 .gitignore 是新文件，完全无视 temp.log 的存在！

    • 使用 git add . (这里的 . 表示添加所有未被忽略的新文件或修改过的文件) 来一次性暂存 styles/main.css 和 .gitignore。

    • 使用 git commit -m "feat: Add styles and gitignore" 提交。

  4. 检查历史:

    • 最后，运行 git log，你应该能看到你刚刚创建的两个提交记录。

完成这个任务后，你就掌握了在本地管理代码版本最核心的技能！
