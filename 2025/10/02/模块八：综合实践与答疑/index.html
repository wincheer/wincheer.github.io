模块八：综合实践与答疑

欢迎来到我们 Git & GitHub 培训的最后一站，也是最重要的一个模块——我们的“毕业项目”！

在过去的七个模块里，我们从零开始，一起探索了版本控制的奇妙世界。你们学会了如何创建本地仓库、如何像玩游戏存档一样管理分支、如何与团队在远程仓库上协作，甚至还掌握了 rebase、stash 这些“老司机”才懂的高级技巧。

现在，是时候把这些散落的“技能点”全部串联起来，完成一次从接手任务到代码上线的完整模拟了。这不再是单个命令的练习，而是对我们公司整个研发工作流的实战演练。

这个模块没有新的命令，也没有复杂的理论。它的目标只有一个：让你自信地走出这个课堂，并且立刻就能无缝地融入到团队的开发节奏中去。把它看作是你们的“第一次飞行”，我会是你们的“塔台”，引导你们完成从“起飞”（创建分支）到“平稳降落”（合并代码）的每一步。准备好，展示你们的学习成果吧！

【场景】

任务背景: 想象一下，今天是你在公司的第一周，你已经熟悉了环境，刚刚，你的项目经理在项目管理工具（比如 JIRA 或 Trello）上给你指派了你的第一个开发任务。

  • 任务卡号: TICKET-101
  • 任务标题: 添加公司简介页面
  • 任务描述: 当前我们的项目 my-remote-practice 只有一个 README.md 文件。现在需要为这个项目创建一个简单的公司简介页面 about.html，介绍公司的基本信息。

核心痛点/目标: 如何将之前学过的所有 Git/GitHub 操作（分支、提交、推送、PR、审查…）组合成一套标准、专业、且符合公司规范的完整流程，来完成这个开发任务？

【理论】

在我们开始敲第一行代码之前，让我们再次鸟瞰整个“开发航线图”。这正是我们采纳的 GitHub Flow 的核心步骤，也是你未来每一天工作的标准节奏：

  1. 同步最新代码 (起飞前检查): 永远确保你的 main 分支是远程仓库最新的状态。
  2. 创建特性分支 (申请新航线): 从最新的 main 分支上，根据任务规范创建一个属于你自己的 feature 分支。
  3. 本地开发与提交 (在航线上平稳飞行): 在你的分支上进行编码、修改、并进行原子化的、有意义的提交（commit）。
  4. 推送远程分支 (向塔台同步你的位置): 将你的本地分支推送到 GitHub，这样你的同事就能看到你的工作进度。
  5. 创建 Pull Request (请求降落): 告诉团队，你的功能已经开发完毕，请求将你的代码合并到 main 分支。
  6. 代码审查与修改 (塔台检查与航线调整): 同事会对你的代码进行审查，你可能需要根据反馈进行修改和再次推送。
  7. 合并与部署 (批准降落，进入主航道): 代码审查通过后，将你的 PR 合并到 main 分支。在真实世界里，这通常会触发自动化部署。
  8. 清理工作 (离开跑道): 合并成功后，删除已经完成使命的 feature 分支，保持仓库整洁。

这就是我们的“飞行手册”，接下来，我们将一步步执行它。

【常见问题 & 资源分享】

1. Q&A 环节

  • “如果我需要处理一个很大的二进制文件（比如设计图、视频），应该怎么办？”
    • 答: 千万不要直接 git add！Git 不擅长处理大型二进制文件的变更。正确的做法是使用 Git LFS (Large File Storage)。它会把大文件存储在专门的服务器上，而在你的 Git 仓库里只保存一个轻量的“指针”。这需要单独安装和配置，但原理上就是“让专业工具做专业的事”。
  • “如果我的 PR 和 main 分支冲突了，该怎么办？”
    • 答: 这是一个非常常见的问题！首先，别慌张。这只是意味着在你开发期间，有人向 main 分支合并了与你修改了同一部分代码的新内容。解决方法如下：
      1. 在本地确保你的 main 分支是最新状态：git switch main 然后 git pull。
      2. 切换回你的特性分支：git switch feature/my-feature-branch。
      3. 将最新的 main 分支 rebase 到你的分支上：git rebase main。
      4. Git 现在会提示你有冲突。打开冲突文件，手动解决（删除 <<< === >>> 并保留正确的代码），然后 git add .。
      5. 继续 rebase 过程：git rebase --continue。
      6. Rebase 成功后，强制推送更新你的远程分支和 PR：git push --force-with-lease。
  • “我不小心把一个包含密码的文件提交到了远程分支，怎么办？”
    • 答: 这是一个紧急情况！
      1. 立刻修改密码！ 这是第一要务。
      2. 从代码层面，你需要从 Git 历史中彻底删除它。这需要使用非常高级和危险的命令，如 git filter-branch 或第三方工具 BFG Repo-Cleaner。这会重写整个项目的历史，需要所有协作者的配合。
      3. 最佳实践是预防: 使用 .gitignore 忽略配置文件，并使用环境变量或专门的密钥管理服务来处理敏感信息。
  • “如何快速定位是谁在哪次提交中修改了某一行代码？”
    • 答: 使用 git blame 命令。它会逐行显示文件内容，并在每一行的行首标注出最后一次修改该行的提交哈希、作者和时间。在 VS Code 等编辑器中，通常有插件（如 GitLens）可以让你更直观地看到这些信息。
