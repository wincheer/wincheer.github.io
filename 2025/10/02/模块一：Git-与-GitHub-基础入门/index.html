模块一：Git 与 GitHub 基础入门

大家好，欢迎参加 Git 新人培训！在接下来的时间里，我会带大家一起掌握一个现代软件开发中最核心、最强大的工具——Git。

请大家暂时忘记它那些看起来复杂又令人生畏的命令。相信我，通过这次培训，你们会发现 Git 其实是一个设计得非常巧妙、能让我们的工作变得无比轻松和安全的“游戏存档”工具。

让我们从一个故事开始吧。

【场景】

想象一下这个场景：你和两位同事，小明和小红，正在共同开发一个重要的项目。你们的工作方式是把代码都放在一个共享文件夹里。

  • 周一：你写了一个登录功能，测试通过后，你把 login.js 文件放进了共享文件夹。

  • 周二：小明为了优化性能，大刀阔斧地修改了 login.js。但他改得太急，引入了一个隐藏的 bug，而且他忘了告诉你他改了什么。

  • 周三：小红为了加一个“忘记密码”的链接，也修改了 login.js。她下载了周二的版本，加入了她的代码，然后覆盖了共享文件夹里的文件。

  • 周四：客户反馈登录功能彻底瘫痪了。

现在，你们三人面面相觑，脑子里充满了问题：

  • 到底是谁最后改了代码？

  • 我们能不能回到周一那个“好的版本”？但当时的代码谁备份了？

  • 小明和小红的代码能不能合并在一起，而不是互相覆盖？

这种混乱、低效且充满风险的协作方式，就是我们今天要解决的核心痛点。我们需要一个系统，能清晰地记录每一次代码变更、允许我们随时回退到任何历史版本、并且能让多人协作开发互不干扰。

这个系统，就是 Git。

【理论】

1. Git 到底是什么？—— 你的终极游戏存档系统

在进入任何命令之前，我们必须先回答这个“第一性问题”：Git 究竟是用来解决什么问题的？

简单来说，Git 是一个“版本控制系统”。

这个名字听起来有点官方，我们换个说法：Git 就是为你的代码项目量身定做的、拥有无限撤销功能和多个平行世界的超级游戏存档系统。

  • 记录每一次变化: 你每完成一个功能，就可以创建一个“存档点”（Commit），Git 会精确记录下你改了哪些文件的哪几行。

  • 随时读档: 电脑蓝屏了？代码改崩了？没关系，你可以轻松地回到任何一个历史存档点，就像读档一样。

  • 开启平行世界: 你想尝试一个可能会失败的新功能吗？你可以开一个“平行世界”（Branch），在里面随便尝试，完全不影响“主线剧情”。成功了，再把它合并回主线；失败了，就直接删掉这个平行世界。

2. Git 的核心工作区——“从草稿到出版”的流程

为了理解 Git 如何工作，我们需要先建立一个宏观的“鸟瞰图”。Git 的核心流程就像一个作家写书并出版的过程。别担心这些新名词，它们非常直观，马上你就会明白——Git 把你的工作分成了四个明确的区域：

  1. 工作区 (Workspace)

    • 比喻: 你的书桌。这是你当前正在进行创作、写草稿、随意修改的地方。你电脑里的项目文件夹就是你的工作区。这里的代码可能是凌乱的、未完成的。

  2. 暂存区 (Staging Area)

    • 比喻: 你放在桌上的**“待出版”文件夹**。当你完成了一章草稿，并且对它很满意，你就会把它放进这个文件夹，准备将它和其他几章一起“存档”。暂存区是一个缓冲区，让你精挑細選，决定哪些修改要被放入下一个“存档点”。

  3. 本地仓库 (Local Repository)

    • 比喻: 你个人电脑里的最终书库。当你把“待出版”文件夹里的所有章节都整理好后，你执行“存档”（Commit）操作，Git 就会为当前所有在暂存区里的文件拍一张快照，生成一个带有唯一编号的版本，永久保存在你的本地仓库里。这个书库包含了你项目的所有历史版本。

  4. 远程仓库 (Remote Repository)

    • 比喻: 云端的中央图书馆，比如 GitHub。你的个人书库虽然安全，但只存在你的电脑上。为了和他人协作，也为了有个更安全的备份，你会把你的本地书库“推送”（Push）到这个中央图书馆。团队里的其他人也可以从这个图书馆“拉取”（Pull）最新的版本到他们自己的电脑上。

小结一下我们的工作流就是： 在书桌上写代码 -> 把写好的部分放进**“待出版”文件夹** -> 将文件夹里的内容出版到个人书库 -> 最后将个人书库的更新同步到中央图书馆。

3. GitHub 是什么？

如果说 Git 是那个强大的存档工具（软件），那么 GitHub 就是一个存放这些“存档”的、基于网络的中央图书馆（网站/服务）。

GitHub 提供了：

  • 远程仓库托管：我们上面提到的“中央图书馆”。

  • 团队协作功能：比如我们后面会学到的代码审查（Code Review）、项目管理等。

  • 开源社区：全球数百万开发者在这里分享和协作他们的项目。

记住：Git 是工具，GitHub 是平台。我们本地用 Git 管理版本，然后通过 Git 把版本同步到 GitHub 上进行协作和备份。

4. 命令行 (CLI) vs 图形界面 (GUI)

  • CLI (Command-Line Interface): 通过在终端（Terminal）里输入 git add . 这样的命令来操作 Git。

  • GUI (Graphical User Interface): 通过 Sourcetree、GitKraken 或 VS Code 插件这样的图形化软件，用鼠标点点点来操作 Git。

我们为什么坚持先学命令行？ 因为命令行是 Git 最直接、最强大的交互方式。所有 GUI 工具的底层，实际上都是在替你执行这些命令行。学会了命令行，你才能真正理解 Git 的工作原理，并在遇到复杂问题时知道如何解决。这就像学开车，先学手动挡，以后开自动挡就游刃有余了。更重要的是，当你遇到图形化工具无法解决的复杂问题时，命令行的知识将是你进行故障排除和深度操作的底气所在。

【核心】

好了，理论说得差不多了，让我们卷起袖子，开始准备我们的开发环境。这个过程非常简单，而且通常只需要做一次。

1. 安装 Git

首先，我们需要检查电脑上是否已经安装了 Git。

友情提示: 打开你的“终端”（在 Mac 上是 Terminal，在 Windows 上可以是 Git Bash 或 PowerShell）。别怕，它只是一个用文字和电脑对话的窗口。

在终端里输入下面的命令然后按回车：

  • 逐行精讲:

    • git: 这是我们和 Git 工具本身打招呼的方式，告诉电脑“我要调用 Git 程序”。

    • --version: 这是我们给 Git 的一个指令，意思是“请告诉我你的版本号”。

如果电脑返回了 git version 2.xx.x 这样的信息，恭喜你，Git 已经安装好了！如果没有，或者提示命令未找到，请访问 Git 官网 下载并安装适合你操作系统的版本。安装过程一路点击“下一步”即可。

2. 基础配置：给自己办一张“作者证”

安装好 Git 后，最重要的一件事，就是告诉 Git 你是谁。这就像你在每一本书的扉页上签上你的名字一样。你未来的每一次“存档”（Commit），都会记录下这个签名。

我们需要配置你的用户名和邮箱。请在终端里依次输入并执行下面两条命令，记得把中文部分换成你自己的信息。

配置用户名:

配置邮箱:

  • 逐行精讲:

    • git config: 告诉 Git 我们要开始进行一些配置。

    • --global: 这个参数的意思是“全局配置”。也就是说，这台电脑上所有的 Git 项目都会默认使用这个配置。这是一劳永逸的设置。

    • user.name "张三": 设定用户名的键值对，键是 user.name，值是你的名字。

    • user.email "zhangsan@example.com": 同理，设定你的邮箱。

为什么要配置这个？ 因为 Git 是一个为协作而生的工具。当你的代码和几十上百人的代码最终合并在一起时，我们必须能清晰地追溯到每一行代码、每一次修改的作者是谁。这个配置就是你的身份标识。

【实践】

现在，轮到你了！这个任务非常简单，只需要花你两分钟，但它能确保你的工具已经准备就绪。

任务目标: 完成你的 Git 基础环境配置。

操作步骤:

  1. 打开终端: 在你的电脑上找到并打开命令行终端。

  2. 检查版本: 输入 git --version 并按回车，确保 Git 已成功安装。

  3. 设置你的名字: 运行 git config --global user.name "Your Name" 命令。请务必使用你希望在工作项目中展示的真实姓名或常用花名（建议使用英文）。

  4. 设置你的邮箱: 运行 git config --global user.email "your.email@company.com" 命令。请务必使用你的公司邮箱。

  5. 检查配置 (可选): 可以运行 git config --list 命令，检查一下你的 user.name 和 user.email 是否已经成功设置。

完成后，你就已经为后续所有 Git 操作打好了坚实的基础！ 在下一个模块中，我们将真正开始创建我们的第一个“代码存档”。
