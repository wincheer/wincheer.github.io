模块三：分支管理与合并

大家好！在之前的模块中，我们学会了如何像一位严谨的作家一样，为我们的项目创建“存档点”。但到目前为止，我们的故事还是一条直线。

真正的软件开发，故事线要复杂得多。想象一下，你正在写书的主线剧情，突然想到了一个绝妙的番外篇，或者编辑突然让你停下手中的活，去修改第一章的一个严重印刷错误。你该怎么办？难道要把主线剧情的稿子弄得一团糟吗？

这就是“分支”将要带给我们的超能力：创造平行世界。在这个模块，我们将学习 Git 最强大、最核心的功能之一——分支。它将彻底改变你组织和思考代码的方式。准备好进入代码的多重宇宙了吗？

【场景】

让我们来面对一个每个开发者都会遇到的经典场景：

你正在公司的主力产品 “Ocean” 项目上，吭哧吭哧地开发一个全新的功能——“用户个人中心”。你预计这个功能至少需要一周才能完成。你已经在代码里写了三天，添加了很多新文件和逻辑。

突然，老板冲到你的座位上，表情严肃：“线上版本出了一个紧急的 Bug！用户无法登录！你必须马上修复它，1 小时内上线！”

现在，请思考这个“第一性问题”： 你要怎么办？

你的代码库里，一半是已经写了三天、还未完成、不能发布的“用户个人中心”功能，另一半是稳定的、正在线上运行的代码。如果你直接在当前的代码上修改 Bug，你发布的时候就会把那堆半成品代码一起发布上去，造成更大的灾难。

难道要把新功能代码一行一行删掉？或者手动复制一份项目文件夹？这些方法都太原始，太容易出错了。

这个进退两难的窘境，正是“分支”要完美解决的核心痛点。

【理论】

1. 什么是分支？—— 轻量级的“平行宇宙”

在 Git 里，分支（Branch）到底是什么？

分支就是从你的主故事线（main 分支）中开辟出的一个“平行宇宙”或“独立时空”。

在这个新的平行宇宙里，你可以自由地进行任何实验、写任何代码，而完全不会影响到主故事线。你可以开发新功能、尝试新的技术方案，甚至把代码改得一团糟都无所谓。当你觉得这个平行宇宙里的故事发展得不错，你可以选择把它“合并”回主故事线。如果觉得不行，直接抛弃这个平行-宇宙即可，主线安然无恙。

从技术上讲，Git 的分支极其轻量，它本质上只是一个指向某个“存档点”（Commit）的可移动的指针。创建一个分支，仅仅是创建了一个新的指针而已，几乎不耗费任何时间和存储空间。

2. HEAD 指针 —— “你当前在这里”的标记

为了理解分支切换，我们必须认识一个新的概念：HEAD。

HEAD 是 Git 中一个特殊的指针，它永远指向你当前所在的分支。 你可以把它想象成游戏地图上那个闪烁的 “You Are Here” 图标。当你切换分支时，实际上就是移动 HEAD 指针，让它指向另一个分支指针。

3. 合并分支 —— 让平行宇宙回归主线

当我们在一个分支上的工作完成后，就需要将它的变更合并回主分支。合并主要有两种类型：

  1. 快进式合并 (Fast-forward Merge)

    • 发生条件: 当你的主分支 main 在你创建功能分支 feature 后，没有任何新的提交，那么 main 分支指针可以直接“快进”到 feature 分支的最新位置。

    • 过程: 不会产生新的“合并存档点”，历史记录是一条直线，非常干净。

    • 比喻: 你写的番外篇大获成功，编辑决定直接把它作为主线剧情的下一章，无缝衔接。

  2. 三方合并 (Three-way Merge)

    • 发生条件: 在你开发 feature 分支的同时，你的同事也在 main 分支上提交了新的更新（比如修复了另一个 Bug）。这时，两条故事线都已经向前发展，Git 无法简单地快进。

    • 过程: Git 会找到两个分支的共同祖先，然后将两边的修改内容进行合并，并自动创建一个新的“合并存档点”（Merge Commit）。这个 commit 很特殊，它有两个父节点。

    • 比喻: 你完成了番外篇，同时主线剧情也有了新的发展。现在需要写一个新章节，把番外篇的人物和主线的新剧情巧妙地融合在一起。这个“融合章节”就是 Merge Commit。

4. 合并冲突 (Merge Conflict) —— Git 的求助信号

好了，现在让我们深吸一口气，来面对每个新人都觉得有点可怕的东西——合并冲突 (Merge Conflict)。别担心，它一点也不神秘。

第一性问题: 为什么会产生冲突？

想象一下，你和你的同事在进行云文档协作。你把书稿中第一章的第三句话改成了“天空是蓝色的。”，而你的同事几乎在同一时间，把同一句话改成了“天空是灰色的。”。当你们要合并文档时，程序就懵了：它不知道该听谁的。

合并冲突就是 Git 在向你发出的求助信号。 它告诉你：“嘿，你在 feature 分支和 main 分支里，修改了同一个文件的同一块区域，我不知道应该保留哪个版本，或者如何将它们组合起来。请你这个人类来当裁判，告诉我最终结果应该是什么样的。”

Git 会在冲突的文件中插入一些特殊的标记（如 <<<<<<<, =======, >>>>>>>），把两个分支的不同内容都展示给你看，等待你手动编辑、解决冲突，然后创建一个新的 commit 来确认最终版本。

【核心】

让我们来完整模拟一次场景引入中的“救火”流程。

假设我们的项目目前只有 main 分支，并且有一个 index.html 文件。

  • 逐行精讲:

    • git switch: 这是较新的、推荐用来切换分支的命令。

    • -c: 是 -create 的缩写，表示如果分支不存在，就创建它。

    • 这行命令等价于两个老命令的组合：git branch feature/PROJ-123-user-profile (创建分支) 和 git switch feature/PROJ-123-user-profile (切换分支)。老手可能还会用 git checkout -b <branch-name>，效果也完全一样。

    小贴士： 旧的 git checkout 命令功能过于繁多（既能切换分支又能恢复文件），为了让命令的职责更清晰，Git 官方在较新的版本中引入了 git switch 命令。我们推荐大家从一开始就养成使用 switch 的好习惯。

现在，我们在这个“平行宇宙”里愉快地开发。修改 index.html，添加新页面等。

紧急情况发生！需要修复线上 Bug！

  • 逐行精讲:

    • git merge hotfix/PROJ-456-login-bug: 告诉 Git：“请把 hotfix/PROJ-456-login-bug 分支上的所有改动，合并到我当前所在的 main 分支来。”

    • 这次很可能会是一次“快进式合并”，因为 main 分支在 hotfix 分支出去后没有动过。

危机解除！现在我们可以安心回到新功能的开发中了。

  • 注意: 这一次合并，很可能会是一次“三方合并”！因为在你开发功能分支的期间，main 分支因为修复了 Bug，已经向前走了一步。Git 会自动创建一个新的 Merge Commit。

如何处理冲突（模拟）

假设在 hotfix 分支和 feature 分支都修改了 index.html 的同一个标题。当你执行 git merge feature/PROJ-123-user-profile 时，Git 会提示 CONFLICT。

打开 index.html，你会看到类似这样的内容：

  • 解读:

    • <<<<<<< HEAD: 这以下，到 ======= 之前，是 HEAD 指向的分支（也就是 main）的内容。

    • =======: 分隔符。

    • >>>>>>> feature/PROJ-123-user-profile: 这以上，到 ======= 之后，是来自 feature/PROJ-123-user-profile 分支的内容。

解决步骤:

  1. 手动编辑: 打开这个文件，删除掉所有 Git 添加的特殊标记 (<<<, ===, >>>)，然后根据你的需求，决定最终代码的样子。比如，你决定都保留：<h1>Login Page - Bug Fixed</h1><h1>User Profile Page</h1>。

  2. 暂存: git add index.html

  3. 提交: git commit (此时 Git 会自动生成一个合并提交信息，你直接保存退出即可，不需要加 -m)。

最后，清理分支 当一个分支的工作已经合并完成，我们应该删除它以保持仓库整洁。

  • git branch -d: -d 是 -delete 的缩写。这是一个安全的操作，如果分支的工作还没有被合并，Git 会阻止你删除。如果想强制删除，使用 -D。

【实践】

是时候让你来体验一下在平行宇宙间穿梭的感觉了！

任务目标: 练习分支的创建、切换、合并，并亲手解决一次合并冲突。

操作步骤:

  1. 准备工作:

    • 创建一个新文件夹 branch-practice 并 cd 进去。

    • git init 初始化仓库。

    • 创建一个 story.txt 文件，内容为 “故事的开头。”。

    • git add . 和 git commit -m "docs: Initial story"。

  2. 开发“好结局”分支:

    • 创建并切换到新分支 feature/good-ending。

    • 在 story.txt 文件末尾添加一行 “王子和公主从此幸福地生活在一起。”。

    • 提交这次修改：git commit -am "feat: Add a happy ending"。代码精讲： 这里的 -a 是 --all 的缩写，它是一个方便的快捷方式，可以跳过 git add 这一步，直接将所有已经被 Git 追踪过的文件的修改添加到暂存区。请注意： -a 无法添加一个全新的（Untracked）文件，对于新文件，你仍然需要先使用 git add。

  3. 开发“坏结局”分支:

    • 切回 main 分支 (git switch main)。这一步至关重要！ 思考一下，如果你不切换回来，而是直接在 feature/good-ending 分支上创建新分支，那么你的“坏结局”就会基于“好结局”的故事线，而不是最初的“故事的开头”，整个分支结构就乱了。我们必须确保每个新功能都从最原始、最稳定的主线出发。**

    • 创建并切换到新分支 feature/bad-ending。

    • 在 story.txt 文件末尾添加一行 “巨龙摧毁了王国。”。

    • 提交这次修改：git commit -am "feat: Add a sad ending"。

  4. 合并与冲突:

    • 切回 main 分支。

    • 首先，合并好结局：git merge feature/good-ending (这应该是一次快进)。

    • 接着，尝试合并坏结局：git merge feature/bad-ending。

    • Bingo！你应该遇到了合并冲突！

  5. 解决冲突:

    • 打开 story.txt 文件。

    • 你会看到 Git 留下来的冲突标记。

    • 请你作为最终的“作者”，决定故事的最终结局。你可以选择保留一个，或者自己写一个新的结局，但务必删除所有 <<<, ===, >>> 标记。

    • 保存文件后，执行 git add story.txt。

    • 执行 git commit 来完成这次合并。

  6. 清理:

    • 删除已经被合并的两个 feature 分支。

完成这个任务，你就掌握了 Git 分支的精髓，真正拥有了驾驭复杂项目开发流程的能力！
